# Structure and wireframes

Turn strategy into an understandable path before choosing visual treatments.

## Define the architecture

1. Map the shortest path from likely entry points to the primary conversion.
2. Create the smallest sitemap that covers audience jobs, proof, objections, and operational pages.
3. Give navigation labels strong information scent; prefer audience language over internal terminology.
4. Assign one primary job, one primary action, and one success condition to each page.
5. Identify shared components and content that should have one canonical source.

## Build text wireframes

Write each page as ordered blocks. For every block, specify:

- purpose and question answered;
- working headline and supporting content;
- required proof or media;
- primary and secondary actions;
- reusable component;
- responsive priority and collapse behavior;
- required interaction and UI states.

Start with the narrow-screen reading order. Expand to wider compositions without changing the message
hierarchy merely to fill space.

## Check the sequence

- Make the page understandable from headings, labels, and calls to action alone.
- Reveal context before details and benefits before feature inventories.
- Place proof near the claim it supports.
- Answer the highest-risk objection before the final conversion request.
- Avoid repeating the same promise in adjacent sections.
- Ensure every action communicates what happens next.

## Define component contracts

For each reusable component, record its purpose, inputs, variants, states, semantic element, responsive
behavior, and acceptance evidence. Include long text, missing optional content, narrow containers, and
keyboard behavior where applicable.

## Gate

Proceed when the sitemap is coherent, the primary path has no dead ends, every section earns its place, and
the narrow-screen reading order preserves the intended persuasion.
