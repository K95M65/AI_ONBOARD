# Authentic product visuals

Treat every visual that represents the product as a factual claim.

## Prefer the real product

Use, in order of preference:

1. A current screenshot or recording of the real product with approved data.
2. The real rendering component populated from a representative safe dataset.
3. A staged visual built from the real schema, labels, formatting, states, and constraints.
4. A clearly conceptual illustration that does not pretend to be a working product view.

Do not build a polished lookalike when an existing product surface can be reused.

## Keep content truthful

- Match commands, labels, ordering, truncation, values, and output format to the real product.
- Make the shown input and result correspond to each other.
- Use realistic lengths, empty values, edge cases, and domain formats.
- Show only capabilities that exist; mark future or conceptual behavior explicitly.
- Use real metrics and customer evidence or label figures as illustrative.
- Never expose production secrets, personal data, or confidential customer information.

Avoid lorem ipsum, invented dashboards, decorative terminals that print marketing copy, fake charts, and
generic AI chrome that implies behavior the product does not have.

## Stage safely when necessary

For pre-release or sensitive products:

- generate representative data from the real schema without copying private records;
- preserve the real component structure and state behavior;
- keep claims conservative;
- label composites, simulations, and illustrative values where users could mistake them for measured output;
- create a repeatable capture path so visuals do not silently drift from the product.

## Validate

Confirm that:

- every visible capability exists;
- every number and label is sourced or explicitly illustrative;
- the visual survives realistic content lengths;
- sensitive information is absent;
- the current product still matches the representation;
- alternative text or surrounding copy communicates the visual's important meaning.
