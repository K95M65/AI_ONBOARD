# The AGENTS.md format

`AGENTS.md` is an open, tool-agnostic standard — think of it as a **README written for AI coding agents
instead of humans**. Humans get `README.md`; agents get `AGENTS.md`. See <https://agents.md>.

## Core properties

- **It's just Markdown.** No required fields, no schema, no frontmatter. If it renders, it's valid.
- **It's tool-agnostic.** Codex, opencode, Cursor, Zed, Jules, Aider and others read it natively; the rest
  can be [wired to it](tool-matrix.md).
- **It's context you pay for every session.** Agents load it on startup, so length is a real cost. Prefer a
  tight, high-signal file over an exhaustive one.

## What to put in it

Include the things an agent can't reliably infer from the code, and the things it will get *wrong* without
guidance:

| Section | Why it earns its place |
|---------|------------------------|
| **Overview** | Orients the agent: what this is, the stack, the shape of the codebase. |
| **Setup** | The exact commands from clone → running. Saves the agent from guessing package managers. |
| **Build / test / lint** | So the agent can verify its own work. Include the *single-test* command. |
| **Structure** | A map of key directories so it doesn't have to spelunk. |
| **How to work** | The behavioral contract — verify-then-claim, persist-within-scope. Ships as-is; see [agent-behavior.md](agent-behavior.md). |
| **Conventions** | Naming, imports, formatting, test patterns — the house style. |
| **Do / don't** | Guardrails: what's off-limits, what's generated, what needs human sign-off. |
| **Gotchas** | Flaky tests, slow builds, required services — the tribal knowledge. |

## What to leave out

- **Anything the code already says.** Don't restate the framework version the agent can read in `package.json`.
- **Tool-specific config.** Hooks, `config.toml`, MCP servers — those live in their own files (see
  [tool-matrix.md](tool-matrix.md#where-tool-specific-power-features-live-keep-these-out-of-agentsmd)).
- **Prose.** Bullets and code blocks beat paragraphs. Agents parse structure well.
- **Aspirational rules nobody follows.** If the codebase doesn't actually obey a convention, don't claim it does.

## Nesting & precedence

Agents read the **closest** `AGENTS.md` to the file they're editing, layering it over ancestors:

```
repo/
├── AGENTS.md              ← applies everywhere
├── packages/
│   ├── api/
│   │   └── AGENTS.md      ← adds/overrides rules for packages/api/**
│   └── web/
│       └── AGENTS.md      ← adds/overrides rules for packages/web/**
```

Rule of thumb: **keep the root file general; push specifics down.** A subdirectory file should contain only
what *differs* from the root — not a copy of it.

Precedence, weakest → strongest:

```
global (~/.codex/AGENTS.md)  <  project root  <  nested subdir  <  the user's live prompt
```

## Style guidance

- Lead with the most important thing (setup + how to run tests).
- Use imperative voice for rules: "Run `npm test` before finishing," not "Tests should be run."
- Fence every command in a code block so agents can copy it exactly.
- Revisit it when it's wrong. A stale `AGENTS.md` is worse than none — it actively misleads.

## Minimal example

```markdown
# AGENTS.md

## Overview
CLI tool in Go that syncs Notion databases to Markdown. Single binary, no server.

## Setup
    go mod download
    make build        # outputs ./bin/notion-sync

## Test
    make test         # full suite
    go test ./sync/   # single package

## Conventions
- Formatting: `gofmt` (CI enforces it — run `make fmt`).
- Errors: wrap with `fmt.Errorf("...: %w", err)`, never swallow.
- No new dependencies without discussion.

## Don't
- Don't edit `internal/gen/**` — it's generated by `make codegen`.
```
